Specification: Flex Menu

Session Type

The flex menu session types clearly define the usage life cycle of flex menu objects. Session types allows a program to further state how it should be used by clients, including human users.
  • INIT = &{new: POPULATE};
  • POPULATE = &{populate: SERVE};
  • SERVE = &{listen: REACT};
  • REACT = +{Event_1: &{action_1: SERVE}, ..., Event_n: &{action_n : SERVE}}}

INIT: This is the start point of the menu's interaction usage life cycle. A client must start by creating a new instance (new).

POPULATE: After creation the flex menu will be ready to be populated. For the client perspective it resumes to calling populate() method; internally it will initialize the menu bar, search and logout buttons, each with their own session type. Respective to it's internal components the flex menu is just another client, hence subjected to use them according to their session types.

SERVE: User interaction components constantly listen to user and system events, and reacts accordingly. When user or system event is fired the object leaves the listening stage and switches on occurred event types

Design: Flex Menu

Structure

The Flex menu is the global user interaction tool with the flex app. The following relationships are true:
  • FlexMenu implements FlexMenuI
  • Flex Menu uses Menu Bar
  • Flex Menu uses Search Button
  • Flex Menu uses Logout Button
The Flex menu delegates user interaction to internal components.
